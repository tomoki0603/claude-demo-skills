# 02 — ワークフローパターン集

## パターン選択ガイド

| ユースケース | 使うパターン |
|------------|------------|
| ファイルに結果を保存する | パターン1 |
| Webページを操作・取得する | パターン2 |
| 条件によって処理を分岐する | パターン3 |
| ユーザーに質問しながら進める | パターン4 |
| 複数ステップを順次実行する | パターン5 |

---

## パターン1: ファイル保存スキル

```markdown
## Step N: ファイル生成と保存

**CRITICAL: 必ず以下のツールを使って保存すること。チャット表示だけで終わらせてはいけない。**

1. **Bash ツールでディレクトリを作成する（存在しない場合）:**
   ```bash
   mkdir -p C:/path/to/output/YYYY-MM
   ```

2. **Write ツールでファイルを保存する:**
   `C:/path/to/output/YYYY-MM/filename-YYYY-MM-DD.md`

3. **Write ツールで index.md を更新する（必要な場合）:**
   `C:/path/to/output/index.md`

### チャット出力
**CRITICAL: 生成した内容はチャットには表示しない。完了通知のみ表示する。**

保存完了後に以下を表示:
```
✅ 生成完了
📁 保存先: output/YYYY-MM/filename-YYYY-MM-DD.md
🕐 生成日時: YYYY-MM-DD HH:MM
```
```

## パターン2: Web操作スキル（Playwright MCP）

```markdown
## Step 1: ページにアクセス

Playwright MCP で `https://example.com` にアクセスする。
ログイン済みセッションを使用する（事前ログインが必要）。

## Step 2: コンテンツ取得

ページのスナップショットを取得し、必要な情報を抽出する。

**CRITICAL: 必ず個別ページを開いて全文を取得すること。**
一覧ページのリード文のみでの処理は禁止。

## Step 3: データ処理

取得したデータを整形・分類する。
詳細な分類基準は `references/[categories].md` を参照。

## Step 4: 保存

パターン1の保存手順に従ってファイルに保存する。
```

## パターン3: 条件分岐スキル

```markdown
## Step 1: 状態チェック

以下の順で確認し、該当するケースへ進む：

### ケース1: [条件A が真の場合]
`[ファイル名/状態]` が存在・成立する場合:
```
「[通知メッセージ]」と表示してスキルを終了する。
```

### ケース2: [条件B が真の場合]
`[ファイル名/状態]` が存在・成立する場合:
→ **Step 2B（[処理名]）** へ進む

### ケース3: デフォルト（いずれも該当しない場合）
→ **Step 2A（通常処理）** へ進む
```

## パターン4: 一問一答インタラクションスキル

```markdown
## CRITICAL: 一問一答ルール
必ず1つの質問だけを行い、回答を待ってから次に進む。

---

## Q1: [最初の質問タイトル]

```
[質問内容をユーザーに表示するテキスト]
（選択肢がある場合は A/B/C 形式で提示）
```

ユーザーの回答を受けて内部で分析し、Q2 へ進む。

---

## Q2: [次の質問タイトル]

Q1の回答を踏まえて:
```
[質問内容]
```

---

## 設計案の提示と承認

Q1〜Qn の回答をもとに設計案を提示する:
```
【設計案】
[設計内容を提示]

この内容でよければ「OK」、修正があれば教えてください。
```

承認後に生成フェーズへ進む。
```

## CRITICALタグの使い方

### 使うべき場面
- ファイル保存（Writeツールの使用）
- 著作権・セキュリティに関わる制約
- 必ず実行すべき後処理（削除・通知）
- 絶対に踏んではいけない処理

### 使ってはいけない場面
- 一般的な手順の説明
- 「推奨」程度のガイドライン
- 全ステップへの乱用（効果が薄れる）

### 正しい例
```markdown
**CRITICAL: 必ずWriteツールでファイルを保存すること。チャット表示だけで終わらせてはいけない。**
```

### 間違った例
```markdown
**CRITICAL: 記事一覧を確認する**  ← 普通の手順にCRITICALは不要
**CRITICAL: タイトルを読む**      ← 乱用
```

## 参照ファイルの設計原則

### 分離すべき内容
- 分類基準・ルール（カテゴリ判定など）
- フォーマットテンプレート
- 用語・組織の解説リスト
- 設定値・しきい値

### SKILL.mdに残す内容
- ワークフローの流れ（Step 1〜N）
- CRITICALな指示
- エラーハンドリング

### 参照の記述方法
```markdown
詳細なカテゴリ基準は `references/01-categories.md` を参照。
```
