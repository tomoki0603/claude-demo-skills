# 03 — description フィールド記述ガイド

## descriptionが重要な理由

`description` フィールドは、Claudeがユーザーの発言を受けて **どのスキルを起動すべきか** を判断するための最重要フィールドである。
適切に書かれていないと、スキルが呼び出されない・別のスキルが起動するなどの問題が発生する。

---

## 基本フォーマット

```
[何をするスキルか、1〜2文で説明]。「[trigger1]」「[trigger2]」「[trigger3]」と言ったときに使用。
```

### 良い例
```yaml
description: 日経電子版の投資関連記事を収集・要約してMarkdownファイルに保存するスキル。「日経をまとめて」「ニュースブリーフィングして」「朝刊を要約して」「日経を実行して」と言ったときに使用。
```

### 悪い例
```yaml
description: 日経ニュースを処理するスキル。
```
❌ トリガーワードがない → Claudeがいつ起動すればいいか判断できない

---

## トリガーワード設計の原則

### 1. ユーザーが実際に使う言葉を使う

ユーザーが自然に話しかける言葉をそのまま列挙する。
技術的な名称ではなく、日常語を優先する。

```yaml
# ✅ 自然な言葉
description: 「コードをレビューして」「PRをチェックして」「コードを見て」と言ったときに使用。

# ❌ 技術的すぎる
description: 「静的解析を実行して」「リンターを走らせて」と言ったときに使用。
```

### 2. スキル名をトリガーに含める

`「[スキル名]を使って」` `「[スキル名]を実行して」` も必ず含める。
ユーザーが直接スキル名で呼び出せるようにする。

```yaml
description: GitHubのIssueを自動作成するスキル。「github-issue-trackerを使って」「Issueを作成して」「バグ報告を登録して」と言ったときに使用。
```

### 3. 3〜6個のトリガーワードを設定する

- 少なすぎる（1〜2個）: 呼び出し漏れのリスク
- 多すぎる（7個以上）: 誤起動のリスク
- **推奨: 4〜5個**

### 4. バリエーションをカバーする

同じ意味でも異なる言い方をカバーする:

```yaml
# 動詞のバリエーション
「〇〇して」「〇〇を実行して」「〇〇を始めて」「〇〇スタート」

# 目的語のバリエーション
「日経をまとめて」「ニュースをまとめて」「今日のニュースを要約して」
```

## カテゴリ別の書き方パターン

### 定期実行スキル（毎日・毎週）

```yaml
description: [サービス名]の[コンテンツ]を収集・要約してファイルに保存する定期実行スキル。「[スキル名]を実行して」「[サービス]をまとめて」「今日の[コンテンツ]を要約して」「[コンテンツ]ブリーフィングして」と言ったときに使用。
```

### コードレビュースキル

```yaml
description: [対象言語/フレームワーク]のコードを[基準]に基づいてレビューするスキル。「コードをレビューして」「[スキル名]を使って」「PRをチェックして」「コードを見て」と言ったときに使用。
```

### データ分析スキル

```yaml
description: [データソース]から[指標]を集計・分析してレポートを生成するスキル。「[スキル名]を実行して」「データを分析して」「[指標]をまとめて」「レポートを作って」と言ったときに使用。
```

### 文書生成スキル

```yaml
description: [入力情報]をもとに[出力文書]を生成するスキル。「[文書名]を作って」「[スキル名]を使って」「[文書名]を生成して」「ドラフトを書いて」と言ったときに使用。
```

## よくある間違いと修正

### ❌ 機能説明だけで終わっている

```yaml
# 悪い例
description: Slackのメッセージを分析して優先度の高いタスクを抽出する。

# 良い例
description: Slackのメッセージを分析して優先度の高いタスクを抽出するスキル。「Slackのタスクを整理して」「slack-task-extractorを実行して」「今日のSlackをまとめて」「タスクを抽出して」と言ったときに使用。
```

### ❌ トリガーが長すぎる・あいまい

```yaml
# 悪い例（長すぎる）
「Slackに届いたメッセージの中からタスクとして対応が必要なものを選び出して優先度をつけてほしい」

# 良い例（短く自然に）
「Slackのタスクを整理して」「Slackをまとめて」
```

### ❌ 他のスキルと被るトリガーがある

複数のスキルが同じトリガーを持つと誤起動の原因になる。
他のスキルのdescriptionを確認して重複を避ける。

```yaml
# 問題: 「まとめて」は汎用的すぎて複数スキルにある
description: 「まとめて」と言ったときに使用。  # ❌ あいまい

# 解決: サービス名・対象を必ず含める
description: 「日経をまとめて」「GitHubをまとめて」と言ったときに使用。  # ✅
```

## チェックリスト

descriptionを書いたら以下を確認する:

- [ ] スキルが何をするか1〜2文で説明されているか
- [ ] トリガーワードが3個以上含まれているか
- [ ] スキル名を使ったトリガーが含まれているか（「[name]を使って」等）
- [ ] ユーザーが自然に言いそうな言葉になっているか
- [ ] 他のスキルとトリガーが重複していないか
- [ ] 「。」で区切って読んだとき意味が通るか
